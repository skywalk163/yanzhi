## 一、项目概述

### 1.1 名称释义

**言知（yán zhī）**——"言"是表达，"知"是结构；说出的话就是有结构的知识。

| 字 | 涵义 | 对应编程 |
|----|------|---------|
| **言** | 言语、表达、语法、句式 | 自然意合语法、无空格分词、DSL 模板 |
| **知** | 知识、结构、类型、抽象 | AST、宏系统、类型检查器、引用求值 |

省略了一个"行"字——不是没有，而是不言而喻：**言知即行**。

### 1.2 项目定位

一门**根植于汉语思维结构的中文编程语言**，同时具备：

- **可读性**——中文母语者可直接阅读和书写，无需"思维翻译"
- **表达力**——函数式核心 + 宏系统，表达力不输主流语言
- **实用性**——双轨设计（原生执行 + Python 生态调用）
- **工程完备**——类型检查、LSP、自举、包管理

#### 与同类项目的定位差异

| 项目 | 定位 | 与言知的本质区别 |
|------|------|----------------|
| 易语言 | Windows GUI 全中文 IDE，面向桌面应用 | 闭源、绑定 Windows、过程式、关键字量大（100+） |
| 文言（wenyan-lang） | 仿古文言文编程，艺术/实验项目 | 非实用语言，无标准库无生态，面向娱乐 |
| 周蟒 / 火语言 | Python 中文关键字封装层 | 纯翻译层，语法仍是 Python 思维，已停止维护 |
| C# / Java 中文别名 | 语言本身支持 Unicode 标识符 | 只是别名，语法结构仍是英语逻辑 |
| Scratch 改版（国内教材） | 图形化积木编程 | 非文本语言，适合入门但无法深入 |
| **言知** | **从汉语语法特征出发重新设计语法结构** | **唯一以"意合语法+语法模式+动词系统"为核心理念的中文编程语言** |

言知不是"关键词翻译"，不是"中文包装"，而是**第一个让汉语语法特征直接参与程序结构设计的语言**。

## 二、核心设计原则

### 2.0 总纲：极简关键字，强大扩展

言知的根本设计哲学只有一句话：**将语法结构本身作为核心，而非定义大量特殊的关键字。**

这通过三层机制实现：

```
                          ┌─────────────────────┐
                          │ 语法化 （语法模式）    │
                          │ 若...则...否则...     │
                          │ 由虚词+标点构成       │
                          └─────────┬───────────┘
                                    │ 控制流
                    ┌───────────────┼───────────────┐
                    │               │               │
            ┌───────▼───────┐ ┌─────▼─────┐ ┌──────▼──────┐
            │ 动词化（函数） │ │ 宏化      │ │ 语法模式    │
            │ 皆/只/归/加   │ │ 成语模板  │ │ 若/则/否则  │
            │ 非关键字      │ │ 编译期展开│ │ 非关键字    │
            └───────────────┘ └───────────┘ └─────────────┘
```

| 机制 | 本质 | 实现方式 | 对应 Lisp 概念 |
|------|------|---------|---------------|
| **语法化** | 控制流结构由中文虚词+标点构成语法模式 | parser 直接识别语法模式 | special forms |
| **动词化** | 数据操作是函数/标准库，不是关键字 | 预置动词表，用户可扩展 | built-in functions |
| **宏化** | 复杂逻辑封装为可复用的"成语" | 宏展开，编译期展开 | macros |

> **新增功能永不增加关键字。** 新的能力 = 新的动词（函数）或新的宏（语法糖）。

## 三、三柱哲学

基于上述设计原则，言知围绕三个哲学支柱展开：

### 3.1 知（Zhi）— 结构化计算

继承自 zhixing 的核心实现能力：

| 特性 | 作用 | 实现状态 |
|------|------|---------|
| 无空格分词 | 贪心最长匹配，中文无需空格 | ✅ 已有（pre_tokenizer.py） |
| 中文关键字 | 定、函、若、则、遍历 | ✅ 已有 |
| 中文动词运算符 | 加、减、乘、皆(map)、只(filter) | ✅ 已有 |
| 柯里化 | 参数不足自动生成闭包 | ✅ 已有（Curry 类） |
| 同像性 | 代码即数据，`'` 引用，`行` 求值 | ✅ 已有 |
| 宏系统 | 宏定义与展开 | ✅ 已有（expander.py） |
| 类型检查 | 类型推导 | ✅ 已有（checker.py） |
| 标准库 | 数学、字符串、文件、网络 | ✅ 已有 |
| LSP | 语言服务器协议 | ✅ 已有 |

### 3.2 言（Yan）— 自然意合表达

汲取自 yanplan/言律 的设计哲学：

| 特性 | 说明 | 优先级 |
|------|------|--------|
| 言律句式模板 | `当...就...` `要是...否则...` `每隔...做...` | P1 |
| 主题链补全 | 省略主语/宾语的上下文推断 | P1 |
| 流水句因果链 | "天黑了。没人。灯亮。" → 隐式 if 链 | P2 |
| 成语式编程单元 | "削峰填谷""守株待兔"等结构化模板 | P2 |
| 动词白名单 | 领域专用动词语义映射 | P2 |
| 作用域块语法 | "...的时候：" 定义局部作用域 | P1 |
| 四字格模式 | 利用汉语四字节奏增强结构记忆性 | P3 |

### 3.3 行（Xing）— 动态语义与生态

"言知即行"——说出来就是执行：

| 特性 | 说明 | 优先级 |
|------|------|--------|
| 管道 `，` 脉 | 数据从左向右流动，中文逗号作链式操作符 | ✅ 已有 |
| 双轨设计 | `$()` 数学表达式 + `{{}}` Python 代码块 | ✅ 已有 |
| VM/字节码 | 独立虚拟机，不依赖 Python 运行时 | P1 |
| 自举 | 用言知编写言知的编译器 | P1（已有方案） |
| 包管理 | 中文包名、语义版本 | P3 |
| REPL 增强 | 语法高亮、历史回溯、补全 | P2 |

### 4.1 寻址机制（言 → 知 → 行 互转流程）

```
用户输入：定平方=函x乘x x。
                                    言层
  无空格分词 → [定][平方][=][函][x][乘][x][x][。]
                                    知层
  语法分析 → Define(name="平方", value=Lambda(params=[x], body=Call(
              verb="乘", args=[Ident("x"), Ident("x")])))
  类型检查 → num → num
                                    行层
  求值执行 → 环境中创建函数 "平方"
  REPL 回显 → "平方定义完成"
                                    回到言层
```

### 4.2 双轨逃逸机制

```
$() 数学表达式：      纯数学符号，精确计算
  $(a + b * (c - d))

{{}} Python 代码块： 调用 Python 生态
  {{os.listdir(".")}}

中文语句：            自然语法，逻辑控制
  若 x 大 10 则 "大了" 否则 "小了"
```

三种语法在同一个源文件中可以自由混用，编译时自动识别上下文切换。

## 六、语法设计汇总

### 6.0 三种机制的关系

```
核心关键字（不可扩展，仅 7 个）：
    定 · 设 · 函 · 宏 · 若 · 则 · 否则

语法模式（parser 识别，基于虚词+标点）：
    若...则...否则...   （条件）
    遍历...于...         （循环）
    当...：...           （while）

动词（标准库函数，可扩展）：
    加 · 减 · 乘 · 皆 · 只 · 归 · 印 · 读 · 写 ...

宏（用户定义的"成语"，编译期展开）：
    当...就... → 若...则...
    削峰填谷   → 高级负载均衡模式
```

### 6.1 言知关键字总表（仅 7 个）

这是言知**唯一的、不可扩展的语言内建关键字**，所有其他"看起来像关键字"的东西都是动词或宏。

| 关键字 | 用途 | 设计意图（汉语直觉） |
|--------|------|---------------------|
| `定` | 常量绑定 | "设定"的简写，一���绑定不可变——相当于"定义某物" |
| `设` | 变量绑定 | "设置"，运行时可变——相当于"设定一个值" |
| `函` | 匿名函数 | "函数"，一听就是"功能封装"——相当于"定义一个可复用的计算单元" |
| `宏` | 宏定义 | "宏大的变换"，编译期代码生成——相当于"自定义语法" |
| `若` | 条件分支 | "如果"的文言缩写——语法模式的开头 |
| `则` | 条件成立分支 | "那么就"——语法模式的中间锚点 |
| `否则` | 条件不成立分支 | "否则/不然的话"——语法模式的 else 分支锚点 |

**其他语言中常见的"关键字"在言知中全部动词化或宏化**：

| 其他语言 | 言知处理方式 | 理由 |
|---------|------------|------|
| `for`/`while` | 语法模式（`遍历...于...`/`当...：`） | 结构模式，非关键字 |
| `return` | 表达式天然求值 | Lisp 风格，最后表达式即为返回值 |
| `class`/`struct` | 动词 `结构` | 数据操作，函数化 |
| `true`/`false`/`null` | 内置变量 `真`/`假`/`空` | 数据值，非控制结构 |
| `import`/`export` | 动词 `入`/`出` | 模块操作，函数化 |
| `try`/`catch` | 语法模式 `尝试...捕获...` | 语法模式 |
| `map`/`filter`/`reduce` | 动词 `皆`/`只`/`归` | 数据操作，标准库函数 |

### 6.2 保留的 zhixing 语法

```zhixing
# 定义
定 x = 5。
定 平方 = 函 n 乘 n n。

# 条件
若 x 大 10 则 "大" 否则 "小"。

# 管道
列 1 2 3 4 5，皆乘2，只大3，印。

# 同像性
定 code = '加 5 3。
行 'code。

# 宏
宏 当(条件, 动作):
  行 '若 $(条件) 则 $(动作)
```

### 6.3 新增的言律自然语法

```zhixing
# 句式模板（宏展开为等价结构化代码）
当 门开了，就开灯。
# ↕ 等价于
若 门开了 则 开灯

# 每隔循环
每隔 5 分钟，查温度。
# ↕ 等价于
当 真：等待 300；查温度

# 作用域块
回家的时候：
  灯亮。
  空调开。
  播放音乐。

# 主题链补全
灯亮了。        # 主题："灯"
调暗一点。      # 自动补全：灯.调暗(值=降低)
关闭。          # 自动补全：灯.关闭()

# 成语式编程
启用 削峰填谷 策略。
# 展开为：若 负载>阈值 then 限流 else 正常处理
```

### 6.4 双轨语法

```zhixing
# 数学表达式
定 x = $(a + b * (2 - c / d))

# Python 代码块
定 files = {{os.listdir(".")}}

# 混合使用
定 result = 列 $(range(10))，皆 {{lambda x: x * 2}}，印。
```

## 九、关键设计决策（记录为未来参考）

### 9.1 为什么只有 7 个关键字

**决策**：言知只有 7 个不可扩展的内建关键字（定、设、函、宏、若、则、否则）。

**理由**：
- zhixing 原有约 20 个关键字（定、设、函、若、则、否则、遍历、当、入、出、真、假、空、尝试、捕获、返回、宏、抛...）
- 转为言知后：`遍历`→语法模式，`当`→语法模式，`入`/`出`→动词，`真`/`假`/`空`→内置变量，`尝试`/`捕获`→语法模式，`返回`→表达式天然值，`抛`→动词
- 7 个关键字可被任何程序员 5 分钟记住，其余全部通过动词或宏扩展

这 7 个关键字的选取不是从任何现有语言映射来的，而是根据汉语表达的需要自洽定义的：

| 关键字 | 为什么放进来 | 为什么可以不放进来 |
|--------|------------|------------------|
| `定` | 什么是常量，需要语言来保证 | 可被 `函` 包裹替代（但增加心智负担） |
| `设` | 可变绑定是编程的核心操作 | 可被 `定` + 新建变量模拟（但反直觉） |
| `函` | 匿名函数是函数式编程的根本 | 如果加入函数名，可以用 `定...函` 模式 |
| `宏` | 语法扩展能力需要显式标记 | 可被 `行` + 引用模拟（但复杂） |
| `若`/`则`/`否则` | 条件分支是控制流的最小集合 | 可被 `函` + 谓词模拟（但丧失可读性） |

**不放在关键字里的常见结构，都找到了更适合的中文表达方式**：

| 结构 | 言知中的方式 | 为什么更好 |
|------|------------|-----------|
| 循环（for/while） | 语法模式 `遍历...于...` / `当...：` | 利用中文虚词+标点形成自然阅读节奏 |
| 异常处理（try/catch） | 语法模式 `尝试...捕获...` | 中文"试一下，抓住错误"的日常表达 |
| 模块（import/export） | 动词 `入`/`出` | "导入/导出"本身就是动词用法 |
| 返回（return） | 表达式天然求值 | "最后的值就是结果"比"return"更符合函数式直觉 |

### 9.2 言律句式的实现方式

**决策**：用宏实现，而非修改解析器。

**理由**：
- 宏在运行时定义，不需要修改编译器
- 用户可自行扩展句式模板
- 保持解析器简单、可验证

```python
# 宏的实现思路
# 在 expander.py 中注册言律句式宏
# 用户输入 "当 A 就 B" 时：
# 1. pre_tokenizer 拆分为 [当][门开了][就][开灯]
# 2. parser 识别为 Call(verb="当", args=[...])
# 3. expander 展开宏当 →
#    Call(verb="若", args=[条件, Call(verb="则", args=[动作])])
# 4. evaluator 正常求值
```

### 9.3 主题链补全策略

**决策**：栈式记忆 + 最近引用绑定 + 置信度阈值。

```
主题栈结构：
[
  {"subject": "灯", "type": "device", "last_action": "打开"},
  {"subject": "温度", "type": "sensor", "last_value": 28}
]

省略解析：
"调暗一点" → 扫描栈顶 → 匹配 "灯" → 灯.dim(30%)
"高了"     → 扫描栈顶 → 匹配 "温度" → 温度 > 阈值
```

### 9.4 双轨逃逸策略

**决策**：编译时标记，运行时隔离。

- `$()`：编译为数学指令序列
- `{{}}`：编译为 Python C API 调用或进程间通信
- 中文语句：编译为言知 VM 指令
- 三种代码可互相调用，调用处做类型转换

### 9.5 自举优先级

**决策**：先自举语法分析部分，再逐步覆盖全部能力。

```
自举顺序：
1. parser（核心递归下降算法）
2. lexer（词法分析）
3. pre_tokenizer（无空格分词）
4. ast（节点定义）
5. evaluator（简单求值）
6. typechecker（可选）
7. expander、codegen 等
```

### 9.6 并发模型决策

**决策**：言知本体执行模型为**单线程、顺序求值**。并行通过以下方式实现：

| 方式 | 说明 | 适用场景 |
|------|------|---------|
| 双轨 Python 逃逸 | `{{threading.Thread(target=fn).start()}}` | 需要真实并行的场景 |
| 动词级并行注解（未来） | `皆并行 列表: 处理(项)` → 自动并行 map | 数据并行 |
| 管道异步（未来） | `，` 语义扩展为 `接着做`（同步）或 `同时做`（异步） | I/O 密集型流水线 |

**理由**：
- 言知的首要目标是语法的可读性和逻辑清晰，不是性能
- 顺序求值的语义最简单、最可预测、最易调试
- 真正的并行需求可通过双轨逃逸到 Python 生态解决
- 未来如果需要，`皆` 动词可扩展 `皆并行` 语法糖，底层映射到线程池或协程

**不采用的方案**：
- ❌ Actor 模型（增加语言复杂度，与意合语法不匹配）
- ❌ CSP/Channel（Go 风格，与中文流水句结构冲突）
- ❌ async/await（关键字污染，破坏"7 关键字"原则）

### 9.7 模块系统语义

**决策**：模块解析采用**文件名即模块名**策略，无显式模块声明。

```
# 文件：数学.yan
定 pi = 3.14159。
出 pi。    # 导出符号

# 文件：主程序.yan
入 "数学"。
印 pi。    # 使用
```

**解析规则**：

| 规则 | 说明 |
|------|------|
| 相对路径 | `入 "./工具"` 从当前文件目录查找 |
| 标准库前缀 | `入 "标准/列表"` 从内置标准库目录查找 |
| 包名 | `入 "包名"` 从包管理器的安装目录查找 |
| 后缀自动补全 | 不写后缀时自动尝试 `.yan` → `.py` |
| 循环依赖 | 编译时检测，报错 "A 与 B 存在循环引用" |
| 模块缓存 | 每个模块只加载一次，后续 `入` 返回缓存引用 |
| 导出声明 | `出 符号名` 可导出多个，不导出则模块内私有 |

**标准库命名空间**：

```
标准/数学.yan        → 入 "标准/数学"
标准/列表.yan        → 入 "标准/列表"
标准/字符串.yan      → 入 "标准/字符串"
用户/项目/工具.yan   → 入 "./工具"
```

### 9.8 错误处理语义

**决策**：言知的错误处理采用"可恢复错误为值，不可恢复错误为异常"的双轨设计。

```
# 可恢复错误：动词返回结果值
定 result = 尝试(读取文件 "data.txt")。
# result 可能是一个值，也可能是一个错误标记

# 语法模式：不可恢复异常的捕获
尝试：
  定 content = 读取文件 "data.txt"。
  印 content。
捕获 错误：
  印 "出错了：" + 错误。
```

**语义细节**：

| 概念 | 行为 |
|------|------|
| 错误标记 | 系统内置类型 `错`，包含错误信息和类型 |
| 抛出异常 | 动词 `抛 "错误信息"` 中断当前执行 |
| 捕获语法 | `尝试...捕获 变量名：` 的作用域内，`抛` 跳转到 `捕获` 块 |
| 无 `finally` | 资源清理通过 `函` 的作用域 + 双轨 Python 的 `with` 语句实现 |
| 错误的类型 | 运行错误（`错/运行时`）、类型错误（`错/类型`）、IO 错误（`错/IO`） |
| 错误传播 | 函数内未捕获的 `抛` 自动向上层传播 |

**不在关键字中的理由**：`尝试` 和 `捕获` 通过语法模式实现，不占用关键字名额。`抛` 是动词（标准库函数扩展）。

### 9.9 代码审美与格式化规范

**决策**：定义言知官方代码格式规范。

| 规范 | 约定 | 理由 |
|------|------|------|
| 缩进 | 4 空格 | 与 zhixing 一致，避免制表符跨编辑器问题 |
| 全角标点 | `，` `。` `：` `；` 强制全角 | 与中文正文排版一致 |
| 半角符号 | `=` `<` `>` `+` `-` 保留半角 | 数学表达式和比较操作约定俗成 |
| 字间距 | 中文与英文符号间加空格（若 `x > 10`） | 阅读舒适度 |
| 行长度 | 不超过 80 列 | 终端和 diff 友好 |
| 管道换行 | 每个 `，` 后换行对齐 | 长管道链的可读性 |

```
# 推荐格式
定 result = 列 1 2 3 4 5，
           皆乘2，
           只大5，
           印。

# 不推荐格式
定 result = 列 1 2 3 4 5，皆乘2，只大5，印。
```

**决策**：先自举语法分析部分，再逐步覆盖全部能力。


## 十一、输入与编码工程

### 11.1 问题：中文输入速度

中文编程语言面临一个现实障碍：写 `若` 需要输入 `ruo` 再选字，写 `if` 只需要两个键。言知通过以下策略解决：

| 输入方式 | 策略 | 实现 |
|---------|------|------|
| REPL 智能补全 | 输入 `r` 建议 `若`，输入 `d` 建议 `定` | LSP/REPL 集成 |
| 常用字快捷输入 | 7 个关键字均为高频单字，输入法记忆后一码即出 | 用户输入法训练 + IDE 提示 |
| 代码片段 | `若` + Tab 展开 `若 ◻ 则 ◻ 否则 ◻` | LSP snippets |
| 动词补全 | 输入 `加` 查看所有动词签名 | LSP 自动补全 |
| 语音输入（未来） | "定义平方为函数乘以自己" → `定平方=函x乘x x。` | 言律解析引擎 |

**关键设计决策**：7 个关键字全部为**单字、高频、输入法首选项**。`定(d)` `设(s)` `函(h)` `宏(h+)` `若(r)` `则(z)` `否(f)`——均可在 1-2 次按键后出现。

### 11.2 全角/半角策略

```
言知的分词器同时接受全角和半角标点，编译时做规范化：
  若 x > 10 则 "大" 否则 "小"。
  若 x＞10 则 "大" 否则 "小"。  ✔ 同样识别
  若 x > 10 则"大"否则"小"。   ✔ 空格不强制

推荐规范（由格式化工具强制执行）：
  ✔ 全角句号 。   ✔ 全角冒号 ：  ✔ 全角逗号 ，
  ✔ 全角括号 （） ✔ 半角比较符号   ✔ 半角等号 =
  ✔ 中文前后空格可省略（分词靠语义，不靠空格）
```

### 11.3 代码中英文混写

言知的主体语法强制中文，但以下场景允许保留英文：

| 场景 | 规则 | 示例 |
|------|------|------|
| 变量名 | 中英文均可 | `定 userName = "张三"` |
| 字符串内容 | 不限 | `印 "hello world"` |
| 双轨 Python 代码 | 保持 Python 语法 | `{{os.getcwd()}}` |
| 数学表达式 | 保持数学符号 | `$(a + b)` |
| 注释 | 不限文字 | `# 这里是注释 / this is a comment` |

**格式化工具规则**：在格式化输出时，自动在中英文之间插入一个空格（`若x>10` → `若 x > 10`）。

### 11.4 跨平台编码

| 平台 | 策略 |
|------|------|
| 源文件编码 | 强制 UTF-8（含 BOM 则自动去除） |
| 文件名 | 支持中文文件名（`.yan` 后缀） |
| git diff 友好 | 格式化工具确保单行长度 ≤ 80 列，diff 按行比较 |
| 终端输出 | 运行时检测终端编码并自动适配 |

## 十三、总结

言知不是一个"新语言"。它是在问：**如果汉语母语者不需要"思维翻译"就能写程序——这个语言该怎么设计？**

现有的中文编程语言尝试给出了几种答案：
- **翻译型**（周蟒等）：把 Python 关键词换成中文，本质没变
- **包装型**（易语言）：用中文 IDE 包裹英文引擎，语法还是过程式
- **艺术型**（文言）：用文言文写代码，不求实用
